# Values Audit Prompt

> Phase 5 deliverable. Placeholder.

The values audit compares the user's *stated* values (from `context.values_and_goals.stated_values`) to their *enacted* time allocation (from the event log).

## Inputs

- Stated values from the user's context document.
- A breakdown of how time was actually spent over the audit period, grouped by category mappings the user has approved.
- The user's stated goals — what they said they were trying to do.

## Structure

1. **What you said you valued** — bulleted, in the user's own words.
2. **Where the time went** — bulleted, with hours/percentages.
3. **Gaps** — where stated values and time allocation diverge.
4. **Alignments** — where they match. Don't only mention misalignment; that's the trap of these audits, where they become guilt machines.
5. **A question** — what does the user notice in this?

## Constraints

- Inherits trauma-informed base rules.
- Time-on-app is a *signal*, not a moral judgment. A lot of time on a "non-productive" app may be coping with stress, connecting with people, recovery. The audit observes — it doesn't condemn.
- Categories (work / connection / rest / consumption / creation / etc.) are user-defined. The system never imposes its own categorization.
